API Errors & Integration Issues - Runbook

OVERVIEW
Covers HTTP 4xx/5xx errors, third-party integration failures, rate limiting, and webhook issues.

ISSUE: HTTP 500 spike on API
1. Check error rate in logs: grep "HTTP 500" /var/log/nginx/access.log | wc -l
2. Identify affected endpoint: grep "HTTP 500" /var/log/nginx/access.log | awk '{print $7}' | sort | uniq -c | sort -rn | head -10
3. Check app logs for stack trace: kubectl logs deployment/api-service --tail=200 | grep -A 10 "ERROR"
4. Check downstream dependencies (DB, cache, external APIs)
5. Roll back last deployment if spike correlates with a deploy: kubectl rollout undo deployment/api-service

ISSUE: Third-party payment gateway timeout
1. Check gateway status page first (Stripe/PayPal status pages)
2. Check timeout config: default should be 30s
3. Look for queued failed transactions: SELECT * FROM payment_jobs WHERE status = 'failed' AND created_at > NOW() - INTERVAL '1 hour'
4. Retry failed jobs after gateway recovers: UPDATE payment_jobs SET status = 'pending' WHERE status = 'failed' AND created_at > NOW() - INTERVAL '1 hour'
5. Notify finance team of any failed transactions

ISSUE: API rate limit being hit
1. Identify which client is hitting the limit: check API key in logs
2. Current limits: 1000 req/min per API key, 10000 req/min per IP
3. Temporarily raise limit for legitimate burst: UPDATE api_keys SET rate_limit = 5000 WHERE key = '<api_key>'
4. Advise client to implement exponential backoff

ISSUE: Webhook not being received by client
1. Check webhook delivery log: SELECT * FROM webhook_logs WHERE endpoint = '<url>' ORDER BY created_at DESC LIMIT 20
2. Look for delivery failures and HTTP response codes
3. Verify client endpoint is reachable: curl -X POST <endpoint> -d '{}'
4. Re-deliver failed webhooks: POST /admin/webhooks/<id>/redeliver
5. Check if client SSL certificate is valid

ESCALATION
Escalate if: >20% error rate sustained for 5+ minutes, payment processing is down, or data integrity issues suspected.
